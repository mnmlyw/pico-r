# Conformance ledger

Gaps found by diffing pico-r's behavior against the official PICO-8 binary
(see `tools/oracle.sh`). Each fixed/confirmed gap has a probe cart in
`probes/` and a golden capture in `golden/`; `cargo test --test conformance`
keeps them from regressing.

## Fixed

| id | subsystem | root cause | fix location |
|---|---|---|---|
| print-shorthand gluing | preprocessor | `?` (print shorthand) was only recognized at true line-start and copied its argument text raw, bypassing every other transform (string-escape repair, bitwise ops, etc). Rewritten as part of the main per-character transform loop so it fires anywhere a bare `?` appears and its argument text gets fully transformed. Confirmed against the oracle that `?`'s args run to the **end of the physical line** unconditionally (comments excepted) — a trailing `end`/`;`/keyword on the same line is swallowed and fails to compile in official PICO-8 too, not a pico-r gap. | `src/preprocessor.rs` (`process_line`) |
| P8SCII control-code string escapes | preprocessor / lexer | `\^`, `\#`, `\-`, `\|`, `\+` inside string literals were falling through the lexer's escape table (`invalid escape` lex error). Verified via oracle probe (`ord(sub(s,i,i))`) that each collapses to exactly one control byte (0x06/0x02/0x03/0x04/0x05) with the following character passed through untouched — the existing `gfx.rs` P8SCII renderer already implements those control bytes, so this was purely a compile-time encoding gap. | `src/preprocessor.rs` (`p8scii_control_escape`) |
| `extract_lhs` string-call sugar | preprocessor | `\` (int-div) and other in-place operators use `extract_lhs` to find their left operand by scanning already-emitted output backward. It didn't recognize Lua's paren-less string-call sugar (`rnd"32"`), so `rnd"32"\1` failed to transform. | `src/preprocessor.rs` (`extract_lhs`) |
| `extract_lhs` unbalanced subscript | preprocessor | Same helper treated bare `[`/`]` as "keep extending" without depth tracking, so `tq[o\64+1]` incorrectly walked back through the *open* `[` into the array name (`tq[o` instead of `o`), corrupting the subscript. Now depth-tracked like `(`/`)`. | `src/preprocessor.rs` (`extract_lhs`) |
| `?` swallowed by expression-boundary scanners | preprocessor | `extract_rhs`/`extract_simple_expr`/`extract_bitwise_rhs` (used by compound-assign and bitwise-op transforms) didn't know `?` always starts a new statement, so e.g. `sum+=0 ?"x"` had the `?"x"` clause folded into the compound-assign's RHS. Added `?` as a hard stop, matching `;`. | `src/preprocessor.rs` (three `extract_*` helpers) |
| `printh` was a no-op | api | Needed for this whole harness to work at all — now writes to stdout on native builds via a shared `display_string` helper (also used by `tostr`). | `src/pico_lua/api.rs` |
| Division by zero returns `inf`/`nan` | interpreter | PICO-8's 16.16 fixed point has no infinity/NaN: `x/0` saturates to `±0x7fff.ffff` (confirmed: `1/0 == 0x7fff.ffff`), sign taken from the dividend; `0/0` saturates positive. `x%0` is always `0` (not derived from the saturated division). Since `\` is preprocessed to `flr(lhs/(rhs))`, this single fix also corrects `5\0` (`->32767`) and `-5\0` (`->-32768`). | `src/pico_lua/interp.rs` (`apply_binop`) |
| Default number-to-string precision | interpreter | PICO-8 displays non-integer numbers rounded to 4 decimal digits with trailing zeros stripped (`1/3 -> "0.3333"`, `2.99998 -> "3"`), not Rust's shortest-round-trip float formatting (which was printing values like `0.3333333333333333`). | `src/pico_lua/value.rs` (`number_to_str`) |
| `tostr()` format flags ignored | api | The second argument to `tostr` was never read. Confirmed exact semantics via oracle: `0x1` = hex `"0xHHHH.LLLL"`, `0x2` = raw 32-bit fixed value as a plain decimal integer, `0x3` = hex digits with no `.` separator. Also fixed `tostr()` with **no** arguments returning `"[nil]"` — confirmed official PICO-8 returns `""` for true no-args, distinct from the explicit `tostr(nil)` case. | `src/pico_lua/api.rs` (`api_tostr`) |
| `extract_lhs` dropped a leading unary minus | preprocessor | `-5\2` rewrote as `-flr(5/(2))` (`-2`) instead of `flr(-5/(2))` (`-3`) — the sign ended up outside the rewritten call instead of inside the divided operand, since `extract_lhs`'s backward scan stopped at `-` unconditionally. Now distinguishes leading unary minus (include it) from binary subtraction (`a-b\c`, correctly stop — the `\`'s lhs is just `b`) using the existing `is_prev_value` helper. | `src/preprocessor.rs` (`extract_lhs`) |
| `chr()` dropped out-of-range args instead of wrapping | api | `chr(256,300,-1)` produced a 0-length string (all three args individually failed a `0..=255` range check and were silently skipped). Confirmed against official PICO-8 that out-of-range ordinals wrap via `% 256` (`chr(256,300,-1)` -> 3-byte string `[0,44,255]`). | `src/pico_lua/api.rs` (`api_chr`) |
| Concat error message text | interpreter | `"a"..nil` raised `"concat nil"`, inconsistent with the "attempt to X a Y value" phrasing used by every other runtime error in the file and not matching official PICO-8's exact text (`"attempt to concatenate a nil value"`). | `src/pico_lua/interp.rs` (`apply_binop`, `Concat` arm) |
| `tonum()` format flags ignored | api | The second argument to `tonum` was never read. Confirmed exact semantics via oracle: `0x1` = hex parse (inverse of `tostr`'s hex format, optional `0x` prefix, optional `.` fractional part), `0x2` = raw fixed-bits reinterpretation (a string is parsed as a plain integer bit pattern; a number uses its own `to_fixed()` representation, which naturally wraps for out-of-range values), `0x4` = return `0` instead of `nil` on parse failure. | `src/pico_lua/api.rs` (`api_tonum`) |
| `pcall`/`error`/string-table removed | api / interpreter | Confirmed (binary + manual) these don't exist in official PICO-8 at all — pico-r had added all three as conveniences. See "Phantom features" below for the full writeup. | `src/pico_lua/api.rs`, `src/pico_lua/interp.rs` |
| `palt(bits)` bit order reversed | api | The single-bitmask form maps bit `k` directly to color `k`. Confirmed against official PICO-8 that bit `(15-k)` controls color `k` (bit 15 = color 0, bit 0 = color 15). | `src/pico_lua/api.rs` (`api_palt`) |
| `pal(_,_,2)` corrupted the draw palette | api | Both the table form (`pal(tbl,2)`) and scalar form (`pal(c0,c1,2)`) only special-cased `p==1` (screen palette); any other `p` including `2` (secondary/pen palette) fell into the draw-palette branch. Confirmed official PICO-8 leaves `0x5f00` untouched for `p==2`; made it a no-op there rather than fabricate the actual secondary-palette storage/effect (not yet implemented — see deferred list). | `src/pico_lua/api.rs` (`api_pal`) |
| `pal({[k]=v})` didn't wrap out-of-range keys | api | Iterated only keys `0..16`; a table with a key outside that range (e.g. `{[20]=5}`) was silently dropped instead of remapping via `key % 16`. Confirmed against official PICO-8. Now enumerates the table's actual entries (via its `HashMap<Key,Value>`) instead of assuming keys 0-15. | `src/pico_lua/api.rs` (`api_pal`) |
| `line()` endpoint wasn't memory-mapped | api / memory | The last `line()` endpoint (used for implicit line-continuation) lived only in Rust struct fields (`PicoState::line_x/line_y`), never written to RAM. Confirmed official PICO-8 exposes it at `0x5f3c`/`0x5f3e` (16-bit each) via `peek2`. Added `ADDR_LINE_X`/`ADDR_LINE_Y` constants and write them alongside the existing struct fields. | `src/pico_lua/api.rs` (`api_line`), `src/memory.rs` |

Corpus impact (188 real BBS carts, 120 frames each, `tests/conformance/sweep_baseline.tsv`):
**91→100 clean, 79→70 load errors, zero regressions** (preprocessor batch). The numbers/math and strings/P8SCII fixes are correctness fixes, not load-error fixes, so they don't move this count further; verified no *real* regressions on top of that baseline (`novaescape-0.p8.png` incidentally went `TIMEOUT -> CLEAN`). One apparent regression, investigated and ruled a test-harness artifact, not a real one: `onek_landscape06-0.p8.png` went `LOAD_ERROR -> TIMEOUT` after the `chr()` fix, because the cart's top-level code has `while(not btn(4)) do ... end` (waiting for the X button) at line 92, previously unreachable because a `chr()` call earlier in the same file crashed first. `run-cart`'s test harness only simulates the O button, not X, so it now hangs on that wait loop — confirmed official PICO-8 itself does **not** hang on this cart (`pico8 -x` exits cleanly). The underlying `chr()` fix is correct and oracle-verified; this is a gap in the sweep harness's input simulation, not in pico-r.

## Deferred / newly discovered (not fixed this batch)

- **`try_bitwise_op`'s RHS splice doesn't recurse.** Unlike compound-assign's RHS (which recursively calls `preprocess()` on its captured span), `try_bitwise_op` copies its captured RHS verbatim from the untransformed source. A backslash nested inside a bitwise RHS (e.g. `e&tq[o\64+1]`, from `picodex_dual-1.p8.png`) still isn't transformed. This is a broader "extracted spans don't recurse" pattern that likely also affects `try_peek_shortcut` — worth its own investigation rather than a one-off patch.
- **`_set_fps` global function missing.** Confirmed via oracle to be a real official PICO-8 API (`pico1kmatch-0.p8.png` calls it at the top level); pico-r has no equivalent.
- **`print()` pixel-level rendering diverges from official PICO-8 even for the trivial case** (`cls(0); print("a")` produces a different screen-byte hash). This is unrelated to the preprocessor work in this batch — likely a default text color, cursor position, or glyph-bitmap difference — and needs its own dedicated investigation before any print-shorthand test can safely assert on screen pixels. All probes in this batch use `printh`/computed values instead of screen hashes to avoid being confounded by it.
- Remaining `LOAD_ERROR` carts in the corpus (70) include a distinct compound-assign bug independently discovered: `obj.rem[axis] = obj.rem[axis] + (axis=="x" and) ox or oy` (`celesteprogrupter-2.p8.png`) — the RHS extraction for `+=` splits a parenthesized `and...or` ternary-style expression incorrectly. Not investigated further this batch.

### numbers/math section audit (architectural, deferred)

- **No 16.16 fixed-point quantization/wraparound in regular arithmetic.** `+`, `-`, `*` (and `/` outside the zero case just fixed) operate as plain `f64` math with no truncation to the 32-bit fixed grid and no overflow wraparound. Confirmed: `32767.5+1` wraps to `-32767.5` on official PICO-8; pico-r gives `32768.5`. `20000*2` similarly wraps (`-25536` on official). This is the single biggest remaining numeric gap — fixing it properly means representing every `Value::Number` as a quantized-after-every-op fixed value (or switching the representation to a raw `i32` entirely), which is a core-VM change, not a local fix. The `to_fixed`/`from_fixed` helpers in `api.rs` (already used by the bitwise ops) are the natural quantization primitives to reuse when this gets tackled.
- **`abs(-32768)` already matches** (`32768` both sides) — the numbers/math audit agent guessed this was broken (predicting `-32768` from wraparound); verified it is NOT a gap. Recorded here so it isn't re-investigated.
- **`sin`/`cos`/`atan2` bypass their own LUTs.** `src/trigtables.rs` defines PICO-8-accurate `SINTABLE`/`ATANTABLE` but nothing calls them — `api_sin`/`api_cos`/`api_atan2` use `std::f64::sin/cos/atan2` directly (after correctly converting PICO-8's "turns" input and flipping sign for the y-down convention, which the audit agent had *also* guessed was broken but isn't). Real gap is precision only: pico-r's trig doesn't reproduce the 4096-entry LUT's quantization steps, so nearby inputs that should land on the same table bucket (and thus produce an identical value on real hardware) instead vary smoothly. Low real-world impact; not fixed this batch.
- **`tostr`'s 4-decimal display doesn't handle exact-tie precision** the same way official PICO-8 does: `tostr(0.12345)` -> pico-r `"0.1235"`, official `"0.1234"`. Same root cause as the fixed-point-quantization gap above — official PICO-8 rounds the *already-fixed-point-quantized* value, and without that quantization step a handful of high-precision inputs round differently at the last digit. Rare in practice (most real carts display values with far less precision); will resolve automatically once the fixed-point epic above is tackled.

### strings/P8SCII section audit

- **`split()` with a multi-character separator string is not literal-substring matching** — confirmed via oracle that `split("a-b_c","-_")` returns `{"a","b_c"}` (2 elements, split only on `-`), not `{"a-b_c"}` (no split, since the literal substring `"-_"` never appears) as pico-r's current `api_split` would produce. The exact rule (charset-style per-character matching? first-character-only?) wasn't pinned down further this batch — needs one more oracle round before fixing. Not yet fixed.
- **`Value::as_number` string coercion requires the whole byte string to be valid UTF-8** (`value.rs`) — a numeric-looking string with an embedded P8SCII byte ≥0x80 (e.g. `"3"..chr(200)`) fails to coerce entirely via `str::from_utf8`, even though only the trailing byte is non-numeric. Not confirmed against the oracle this batch (needs a probe); flagged from code reading only.

#### Phantom features: pico-r had functions/methods that don't exist in official PICO-8 at all (removed)

Not "gaps" in the usual sense (missing behavior) — the reverse: pico-r was accepting code that would **crash on real hardware**. Confirmed two ways: (1) via oracle (`tostr(pcall)`, `tostr(error)`, `tostr(string)` are all `[nil]`/errors across cart-version headers 8/16/27/32/41/42 on the actual installed binary, v0.2.7 per `Info.plist`, ruling out a version-gating explanation), and (2) cross-referenced against the official manual (fetched independently): no mention of `pcall`, no `error()` function, no `string` library — only standalone functions like `sub()`/`chr()`/`ord()`, never `string.sub()`. Removed:

- **`pcall`/`error` globals** — deleted `lua_pcall`/`lua_error` and their registrations entirely (`api.rs`).
- **The `string` table and colon-method dispatch on strings** — deleted the `string` table registration and its `str_len`/`str_lower`/`str_upper`/`str_rep`/`str_format` helpers, and removed `table_get`'s (`interp.rs`) fallback that routed `s:method()` calls to it. Also tightened `MethodCall`'s error to say `"attempt to call method 'X' (a Y value)"` — matching official's exact phrasing — instead of the generic `call_value` message, for any non-callable method lookup (not just strings).
- **NOT a phantom feature** (verified, ruling out a guess, kept as-is): bracket-indexing a string (`("hello")[1]`) returns `"h"` on **both** official PICO-8 and pico-r — this is real, matching behavior.

Verified zero regressions on the 188-cart corpus after removal (confirming no real cart relied on these) beyond the already-documented `onek_landscape06-0.p8.png` harness artifact above.

### gfx draw-state section audit

- **`fillp()` DOES apply to `oval()`/`ovalfill()`** — a wiki claim (that the fill pattern list excludes ovals) was checked against the oracle and refuted: fillp'd and non-fillp'd ovalfill hashes differ on official PICO-8 just as they do without fillp being oval-specific. Recorded so it isn't re-investigated as a gap.
- **`ovalfill()`'s baseline (no-fillp) screen hash doesn't match official PICO-8 either** — found while checking the fillp claim above. This is a genuine, separate rasterization-precision gap (not a draw-state issue), handed off to the upcoming gfx-rasterization section (task #12) rather than investigated here.
- **`circfill()`'s invert fill triggers from the wrong condition.** `poke(0x5f34,0x2)` alone has zero effect on official PICO-8 (confirmed: hash identical to a non-inverted circfill) — invert apparently also requires specific bits (reportedly `0x1800`, per forum text, not yet oracle-verified) set in the *per-call color argument*, which pico-r doesn't look at at all (it inverts from the `0x5f34` register alone, and pico-r's hash *does* differ from its own non-inverted baseline — a real, confirmed-broken trigger condition). Fixing this needs the exact color-argument bit-encoding pinned down first; not done this batch.
- **`pal(_,_,2)`'s actual secondary-palette storage/effect is still unimplemented** — only fixed the confirmed part (it must not touch `0x5f00`). Where the "pen"/secondary palette actually lives in memory and how it's consumed by fillp-based dithering wasn't investigated further; currently a safe no-op rather than a full implementation.
- **`print()`'s cursor-x register (`0x5f26`) is never written back after printing, and the `\n` reset value should come from `0x5f24`** (a documented "margin" register), not the original call's `x`. Found while probing draw-state registers, but this is really a print/font-subsystem concern — deferred to that section (task #13) rather than partially fixed here.

### gfx rasterization section audit

- **`draw_oval` had a real, high-impact algorithm bug — fixed.** The old single-loop incremental midpoint-ellipse implementation could terminate before its stepping variable reached zero, entirely missing the top/bottom pole pixels — confirmed even for an exact-integer bounding box (not just an odd one). Replaced with a per-row (and, for outlines, also per-column) exact scan directly from the ellipse equation, which can't have that class of bug. Two conventions had to be reverse-engineered empirically (row-by-row/pixel-by-pixel diffing against the oracle, not guessed): (1) `x0..=x1`/`y0..=y1` are inclusive pixel ranges, so the *continuous* radius is `(x1-x0+1)/2` — using `(x1-x0)/2` shifts the whole curve and breaks top/bottom symmetry; (2) sampling must use pixel *centers* (`p+0.5`) on both axes. `ovalfill` now matches official pixel-for-pixel for every size tested except one (below); `oval` (outline) needed a **second**, column-based scan for the near-pole regions, where the curve is nearly horizontal and official fills a short flat run of pixels rather than just the two boundary columns — matches exactly for every size tested.
- **One residual, unresolved `ovalfill` mismatch**: for a 80x60 bounding box specifically, 4 of 61 rows (symmetric pairs, e.g. row 12/68 and 18/62) are off by exactly 1px on each side. Confirmed NOT a floating-point precision artifact (`f32` gives the identical result to `f64` at these rows). Most likely official's real rasterizer is itself an incremental algorithm with its own discretization quirks at specific boundary cases, rather than the exact-continuous-math this fix uses — replicating it bit-for-bit would mean reverse-engineering an undocumented, closed-source algorithm further, which weighed against the value of the last few pixels wasn't done this batch. 57/61 rows match exactly (this was previously ~0/61 correct beyond luck).
- **`circ()`/`circfill()`/`line()` (Bresenham) are already pixel-exact** — verified via full hash + spot-pixel checks (a community forum claim of a small-radius "rightward skew" in native `circ()` was checked and refuted for the radii tested). No work needed; recorded so these aren't re-investigated.
- **`sspr()`'s scaling has a confirmed, unresolved rounding mismatch.** Verified via a full pixel-grid diff (not just a hash) that pico-r's `px*sw/dw` nearest-neighbor source-index formula produces a slightly different (shifted-by-one-ish) sampling pattern than official for non-integer scale ratios (tested 8x8 -> 25x25). Tried the analogous "pixel-center" fix that resolved the oval case (`(px+0.5)*sw/dw`, and a "map destination center back to source" variant) — neither matched official exactly. Needs more oracle samples to pin down the exact formula; not fixed this batch.

Corpus regression sweep: zero new regressions (same already-documented `onek_landscape06-0.p8.png` harness artifact).

### print/font section audit — resolved the session's original `print()` mystery

- **`src/gfx_font.rs`'s entire font table was wrong, and this was the root cause of the very first divergence noticed in this whole audit** (`cls(0);print("a")` producing a different screen hash than official PICO-8, flagged as unexplained in the first commit's ledger entry). Root cause: pico-r's font hard-coded **the same bitmap for uppercase and lowercase letters** (`data[(letter+32)] = data[letter]`), but official PICO-8 renders uppercase A-Z as a distinct "small caps" glyph set (row 0 always blank, glyph compressed into rows 1-5) — a real, different bitmap, not lowercase shifted. Confirmed by bulk-dumping all 224 printable P8SCII glyphs (32-255) directly from the official binary via a single oracle probe (`tests/conformance/probes/font_glyphs.p8` regression-locks a representative sample) and mechanically regenerating `FONT_DATA` from that dump — **196 of 224 glyphs were wrong**, not just the upper/lower case ones. `cls(0);print("a")`'s screen hash now matches official exactly (`23296`), as does a full mixed-case/digit/symbol string and the two full alphabets. Zero regressions on the corpus despite this touching every text-rendering cart.
- **`print()`'s return value only reflected the last line's x, not the max across all lines.** Confirmed against official: `print("hello\nb")` returns the x reached after `"hello"` (the longer first line), not after `"b"`. `draw_text` now tracks a running max. | `src/gfx.rs` (`draw_text`)
- **`print()` never persisted the cursor registers when explicit x/y/col were given, and even the two argument forms that did update them only advanced `cursor_y` by a flat `+6` regardless of embedded newlines.** Confirmed against official: cursor_x always resets to *this call's* starting x (not wherever printing ended up, and not the previous register value) and cursor_y advances by `6 * (1 + newline count)`. All three `print()` argument forms now persist both registers uniformly. | `src/pico_lua/api.rs` (`api_print`)
- **The `0x5f24` "margin" register (deferred from the gfx draw-state section) is now precisely characterized as a separate, narrower, still-open gap**: `cursor(x,y)` and an internal `\n` normally reset to the same x, so the fix above already covers ordinary usage — but a cart that explicitly `poke(0x5f24, n)`s a *different* value still diverges (confirmed: `cursor(20,20); poke(0x5f24,60); print("a\nb")` -> official's cursor_x ends at `60`, pico-r's at `20`). Fixing this fully means threading a real persistent margin register through `draw_text`'s `\n`/reset handling instead of using the call's literal `start_x`; not done this batch given how rarely a cart would diverge the two on purpose.
- **Found but not oracle-verified this batch** (code-reading only, from `gfx.rs`'s control-code handling): control code `0x02` takes 3 parameters (offset-x, offset-y, char) where the manual describes a 1-parameter "solid background colour" code; `0x03`/`0x04`/`0x05` treat their parameter byte as signed (`i8`, -128..127) where the manual's `P0-16` framing implies a different bias/range; `0x0E`/`0x0F` are implemented as char-width toggles but the manual describes them as custom-font-switch codes; control code `0x0B` (`\v`, "decorate previous character") has no handler at all. Also: auto-scroll when printing past the bottom of the screen appears entirely unimplemented (`grep -rn scroll src/` is empty), and the custom font at memory `0x5600` isn't wired up (`draw_char` always reads the static table, never memory) — both real feature gaps, not yet oracle-verified for exact behavior.

### map/sprites section audit

- **The big-map width register (`0x5f57`) had its zero-case backwards — fixed.** pico-r treated a raw `0` in this register as width 128 (and never initialized the register at boot, so it defaulted to `0` and happened to hit that branch by coincidence). Confirmed against official PICO-8 that `0` actually means width **256** (`poke(0x5f57,0); mset(200,10,7); mget(200,10)` -> `7` on official, `0` on pico-r pre-fix). Since flipping the interpretation alone would have broken the normal 128-wide default, `Memory::new`/`init_draw_state` now explicitly write the byte `128` at boot instead of relying on the old (wrong) zero-special-case — verified the plain default-width case still behaves identically before and after. | `src/gfx.rs` (`map`/`map_get_wide`/`map_set_wide`), `src/memory.rs`
- **`fget`/`fset` out-of-range index handling already matches** — verified via oracle (`fget(-1)` and `fget(9999)` both return `0` on official PICO-8, same as pico-r's existing clamp). No fix needed; recorded so this isn't re-investigated.
- **Two candidates from this section's audit were empirically inconclusive, not confirmed either way** — flagging rather than guessing:
  - Whether `0x5f56` (the big-map base-address register) is meant to also remap the *normal* (`x<128`) map region, not just the wide-extension columns: a probe poking `0x5f56=0x10` then `mset(0,0,9)` showed **neither** `peek(0x1000)` nor `peek(0x2000)` becoming `9` on official PICO-8 — contradicting both pico-r's current behavior (always writes `0x2000`) and the hypothesis that it should write `0x1000`. The write evidently lands somewhere else this probe didn't check, or `0x10` isn't a value official interprets the way assumed. Needs a properly designed follow-up probe before touching this code.
  - The shared sprite/map memory aliasing (sprite rows 64-127 and map rows 32-63 sharing physical bytes at `0x1000-0x1FFF`): pico-r's `sset(0,64,5)` visibly affects `mget(0,32)` (reads back `5`), demonstrating pico-r's own storage *is* shared as designed — but the identical probe on official PICO-8 shows `mget(0,32)` staying `0`, meaning either the exact row/nibble-offset arithmetic assumed for this probe is wrong, or the real aliasing works differently than modeled. Not chased further this batch; needs a probe that empirically discovers the correct address formula (e.g. sweep `poke()` at known addresses and read back via both `sget`/`mget`) rather than assuming one.

Corpus regression sweep: zero new regressions (same already-documented `onek_landscape06-0.p8.png` harness artifact).

### memory/registers section audit

- **`cstore()` didn't protect the compiled-code region (`>=0x4300`) — fixed.** Confirmed against official PICO-8 that `cstore()` silently *skips* writes at or beyond `0x4300` rather than writing through or erroring — verified precisely with a destination range straddling the boundary (`cstore(0x42fe,0,4)`: the two bytes landing at `0x42fe`/`0x42ff` write through, the two at `0x4300`/`0x4301` are silently dropped). pico-r wrote through unconditionally. | `src/pico_lua/api.rs` (`api_cstore`)
- **Memory addresses didn't use `flr()` semantics for negative fractional values — fixed.** Confirmed `peek(-0.5)` reads the same byte as `peek(-1)` on official PICO-8 (both wrap to `0xffff`), not address `0` (which truncate-toward-zero, `as i32`, would give for `-0.5`). Added a dedicated `arg_addr` helper (floor before cast) and applied it to the address arguments of `peek`/`poke`/`peek2`/`poke2`/`peek4`/`poke4`/`memcpy`/`memset` — deliberately scoped to address parsing only, not the shared `arg_int`/`safe_to_i32` helpers used pervasively elsewhere for unrelated arguments that haven't been verified to need the same treatment. | `src/pico_lua/api.rs`
- **Two candidates were oracle-tested and refuted** (already matching official, no fix needed): `fget`/`fset` out-of-range indices already return `0` like official; `memcpy` with a negative length or a destination range extending past `0xffff` already no-ops/clamps like official (doesn't wrap and corrupt low addresses, and doesn't perform a huge accidental copy) — both had been flagged as *suspected* bugs from code-reading alone, refuted only once actually run against the oracle.
- **`stat(0)`/`stat(1)` (memory/CPU usage) are hardcoded constants**, not tracking real heap/frame-time usage — a known, pre-existing stub. Real carts occasionally branch on these for debug overlays; left as-is since replicating PICO-8's exact profiling isn't practical, and this is really `system/stat`-section territory (task #16) rather than a memory-access bug.
- **Tooling fix, not a pico-r conformance bug**: discovered that running a `cstore()`-using probe through `tools/oracle.sh` caused the *official binary itself* to rewrite the cart file back to disk on exit (version bump 42→43, ~64 lines of default `__sfx__` boilerplate appended) — official PICO-8 treats a cart that called `cstore()` as "dirty" and auto-saves it, even under `-x`. This was mutating the *committed probe file* in place, since oracle.sh ran the binary directly on it. Fixed `oracle.sh` to copy the probe into the scratch `-home` dir first and run the copy, so the committed source can never be touched by this side effect (verified: previously-committed probes were all unaffected since none of them exercised `cstore()` before this section).

### system/stat section audit

- **`stat(16..19)`/`stat(20..23)` were swapped — fixed.** Confirmed against official PICO-8: `stat(16..19)` is the per-channel *sfx index*, `stat(20..23)` is the *note index* — pico-r's two branches had these reversed (verified with `sfx(3,0); stat(16)` -> `3` officially, was returning the note index instead). | `src/pico_lua/api.rs` (`api_stat`)
- **`stat(100)` (breadcrumb label) now returns `nil` when unset** instead of falling through to the generic `0` every other unhandled `stat()` code defaults to — confirmed via `type(stat(100))`. | `src/pico_lua/api.rs` (`api_stat`)
- **`cartdata()`/`dget`/`dset` don't actually persist across sessions — a real, confirmed, but larger gap, not fixed this batch.** Verified official PICO-8 durably persists cartdata: writing via `dset` in one process and reading via `dget` in a *separate* `pico8 -x` invocation (same `-home`, simulating a fresh cart load) correctly returns the persisted value. pico-r's `cartdata()` is a pure no-op and `dget`/`dset` only read/write RAM for the current run — the only persistence mechanism anywhere in the codebase is the manual, single global save-slot triggered by pressing P/L in the web player (`lib.rs`'s `web_save_state`/`web_load_state`, unrelated to `cartdata()` and not scoped per-cart-id). Unlike this batch's other fixes, implementing this properly means adding new WASM↔JS boundary exports so the browser can back per-cart-id `dset`/`dget` calls with `localStorage` automatically — a genuinely different scale of change than a local Rust bug fix, and the native `run-cart` CLI used for testing has no equivalent concept of "persistent storage" to build against without extra tooling. Flagged for a dedicated future pass rather than attempted here.
- **`menuitem()` and `extcmd()` are fully inert** (no pause-menu system exists to hook into; `extcmd`'s documented actions like `"screenshot"`/`"video"`/`"pause"`/`"reset"` are all no-ops) — real gaps, but not practically checkable via a headless `printh`-based probe (they're GUI/host-integration features), so not oracle-verified or fixed this batch.
- **`stat(4)` (clipboard contents) and `stat(31)` (last keyboard character) are hardcoded empty strings**, and `stat(6)` (launch parameter string) is likewise always empty — all three require live input/host-environment state this session's headless oracle probes can't simulate, so left unverified rather than guessed at.

Corpus regression sweep: zero new regressions (same already-documented `onek_landscape06-0.p8.png` harness artifact).

### audio API section audit

- **`sfx(id, ch, offset)` with an out-of-range offset didn't clamp — fixed.** Confirmed against official PICO-8 that an offset outside `0..32` clamps to the nearest valid note (`sfx(5,0,99)` starts at note `31`, the max), it doesn't fall back to note `0`. | `src/audio.rs` (`play_sfx`)
- **`music(pattern, fade, mask)` conflated an explicit `mask=0` with an omitted mask — fixed.** Both previously collapsed to the same "reserve all 4 channels" default (`if mask > 0 { mask } else { 0xF }`), but confirmed against official PICO-8 that an *explicit* `music(0,0,0)` reserves **zero** channels, leaving all 4 free for `sfx()` — verified by checking all four `stat(16-19)` channel states, not just one. Fixed by giving the "omitted" case its own sentinel (`-1`) distinct from an explicit `0`, so only a truly-omitted mask still defaults to all channels reserved. | `src/audio.rs` (`play_music`), `src/pico_lua/api.rs` (`api_music`)
- **`sfx(id, -2)`'s exact immediate-vs-release-loop stop semantics are confirmed to diverge, but deliberately NOT changed this batch — too risky without stronger evidence.** A probe checking `stat()` immediately after `sfx(5,-2)` shows official reports the channel as stopped right away, while pico-r's current `release_loop` only marks the loop as released (letting the current pass/tail finish, `finished` stays `false` until the audio naturally ends). The existing code has a comment explicitly citing this exact release-instead-of-stop behavior as a deliberate fix for a real cart (`mansion_bros`' vacuum SFX) — changing it based on a single immediately-after-the-call `stat()` snapshot risks regressing that already-fixed case if the real distinction is "stat reports stopped immediately, but audio synthesis still tails off," which this session's tooling can't distinguish from "actually stops instantly" without recording actual audio samples over time. Needs dedicated investigation (and probably a way to observe the audio engine's internal state over several frames, not just `stat()` once) before touching this.
- **The default channel-reservation behavior when `music()`'s mask argument is omitted entirely couldn't be conclusively tested.** A probe comparing `music(0)` (no mask) against explicit `music(0,0,15)` was ambiguous because this session's probe carts have no real `__music__` pattern data — with an empty/silent pattern, there's no way to distinguish "channel is reserved but produces no sound" from "channel was never actually reserved," so `sfx()` could freely use channel 0 in both cases regardless of which behavior is correct. pico-r's existing default (omitted mask reserves all 4 channels) was left unchanged since there's no counter-evidence, but it's genuinely unverified — a follow-up would need a probe cart with real per-channel music pattern data.
- **`sfx-auto-channel-ignores-music-reservation`** (the audit's top-priority candidate) also couldn't be conclusively tested for the same reason (no real music pattern data to observe genuine channel contention) — deferred alongside the item above rather than guessed at.

### Lua/preprocessor semantics section audit (final section)

- **`del()` during `all()`-iteration could skip or double-visit items — fixed.** The iterator only tracked "did the table's length shrink since last call" to decide whether to advance, which can't distinguish *where* a deletion happened: deleting a *later* item also shrinks the length but leaves the current position's content unchanged (must still advance normally), while deleting the *current* item shifts a later item down into this same slot (must re-visit it) — confirmed against official PICO-8 that the two cases genuinely differ (`del(objs,4)` from inside a loop currently at `v==2` must NOT cause `2` to be re-visited, but pico-r did). Fixed by tracking the last *value* returned (not just the index) and re-checking whether that position still holds it, rather than inferring from length alone. Verified this doesn't regress the already-working "delete the current item" and "delete every even item" patterns. **Confirmed real-world impact**: fixed a previously-crashing corpus cart (`shaiza-1.p8.png`: `UPDATE_ERROR -> CLEAN`) on top of the usual zero-regressions sweep. | `src/pico_lua/api.rs` (`api_all`)
- **Arithmetic and concat operators ignored table metatables entirely — fixed for `__add`/`__sub`/`__mul`/`__div`/`__mod`/`__pow`/`__concat`.** Confirmed against official PICO-8 (whose own manual has a worked example redefining `+` via `__add` for 2D vector math) that these ARE real, working metamethods — unlike `pcall`/`error`/the string library earlier this session, this is core language behavior PICO-8 does support. `apply_binop` previously called `as_number()`/`as_str()` directly with no metatable lookup, so e.g. `vector1 + vector2` crashed with `"arith on table"` instead of dispatching to `__add`. Fixed by checking each operand's metatable (operand `a` first, then `b`, matching standard Lua precedence) before falling back to the existing raw-value logic — verified `__add`/`__sub`/`__mul`/`__concat` all dispatch correctly and non-table operands are completely unaffected. | `src/pico_lua/interp.rs` (`apply_binop`, new `metamethod` helper)
- **Found but not fixed this batch, lower priority / more invasive** (each would need touching a different, more scattered call site than the contained `apply_binop` fix above): `__eq`/`__lt`/`__le` (table comparison) are ignored by `values_equal`/`compare`; `__newindex` is ignored by `table_set` (writes always land directly, no proxy/validation hook); `__len` is ignored by the `#` unary operator (always uses `raw_len()`); `__tostring` is ignored by `tostring()`/implicit string coercion (tables always print as `"[table]"`/an address-style placeholder). None of these were oracle-verified this batch (unlike `__add` etc., which were) — flagged from code-reading and standard Lua 5.2 semantics only, so treat as *plausible*, not confirmed, until checked against the oracle.
- **`pairs()`/`next()`'s key order is Rust `HashMap` order (randomized per process), not Lua's own unspecified-but-stable-per-build order.** A cart iterating a hash-keyed table with `pairs()` and relying on *consistent* (if arbitrary) ordering across runs — e.g. for seeding a deterministic-looking layout — could see different behavior between pico-r launches, and possibly between pico-r and official PICO-8's own (different but consistent) internal order. Real PICO-8's exact order isn't grounded anywhere accessible this session; flagged as a known source of divergence for hash-part iteration, not fixed.

Corpus regression sweep: zero new regressions (same already-documented `onek_landscape06-0.p8.png` harness artifact); two improvements, including the `shaiza-1.p8.png` fix noted above.

## Overall corpus impact across all ten sections

Starting from the harness-bootstrap baseline (91/188 clean), the ten section audits above collectively fixed real, oracle-confirmed gaps across the preprocessor, number formatting, string handling, three graphics subsystems, memory, system/stat, audio, and core Lua semantics — with **zero net regressions** at every checkpoint along the way (the one recurring "regression," `onek_landscape06-0.p8.png`, was investigated once and traced to a test-harness input-simulation limitation, not a real behavior change). A full re-sweep after all ten sections puts the corpus at **102/188 clean (up from 91/188)** — two carts were confirmed fixed outright by name (`novaescape-0.p8.png`, `shaiza-1.p8.png`), the rest of the net gain came from the preprocessor batch's load-error fixes. `tests/conformance/sweep_baseline.tsv` reflects this current state. The very first bug found (a broken font table) turned out to be the root cause of the divergence that motivated this whole audit in the first place. Every fix above has a standing regression probe in `tests/conformance/probes/` with a golden captured from the official binary, so `cargo test --test conformance` keeps all of this locked in going forward.

### Follow-up: compound-assign `and`/`or` splitting bug (post-audit)

After the ten sections above, went back to a bug flagged early in this session but not yet fixed: `extract_rhs` (used by compound-assign, e.g. `+=`) has a heuristic that stops capturing the right-hand side when it sees "a value, then a space, then what looks like the start of a new statement" — but it never checked whether the *word immediately before that space* was itself `and`/`or`/`not`. So a very common PICO-8 idiom, `x+=cond and a or b`, was splitting right after `and`, producing `x = x + (cond and) a or b` — leaving `a or b` dangling outside the parens as malformed code. Confirmed on a real corpus cart (`celesteprogrupter-2.p8.png`: `obj.rem[axis]+=axis=="x" and ox or oy`). Fixed by checking the preceding word against the same `is_operator_keyword` check already used for the *following* word; verified the original heuristic's actual purpose (detecting a genuine same-line statement boundary like `x+=1 y=2`) is unaffected. | `src/preprocessor.rs` (`extract_rhs`)

Corpus re-sweep: **103/188 clean** (up from 102), one full fix (`witchcrafttd-6.p8.png`: `LOAD_ERROR -> CLEAN`) and three carts progressing further (`to_take_root_among_the_stars_1-7.p8.png`, `tomorrow-6.p8.png`, `ultimate_selfie-2.p8.png`). Two carts show as "regressions" by the coarse rank metric (`celesteprogrupter-2.p8.png`/`-3.p8.png`: `LOAD_ERROR -> TIMEOUT`) but are **not real regressions** — investigated and confirmed this is the third occurrence this session of the same pattern (see the `chr()`/`onek_landscape06-0.p8.png` note above): removing the parse error that used to block these carts early now lets their top-level code reach `flip() goto _lp` — an old-style Celeste-derivative game loop that needs real coroutine/`flip()` support pico-r doesn't have (a known, pre-existing, documented limitation, not something this fix caused or could reasonably fix).
